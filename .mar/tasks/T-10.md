Symptom: browsing mar is sometimes very slow; Chrome shows requests "stalled".
HAR proof: GET /board total 43,291ms with blocked=43,273ms but server
wait/TTFB=6.8ms; GET /favicon.ico total 35,755ms, blocked=35,750ms, TTFB=2ms.
The server is microsecond-fast — the time is 100% "blocked" (waiting for a
connection slot), not server latency.

Root cause: layout.gohtml opens a permanent EventSource("/events") on every
page; the handler holds it open forever. mar serves plain HTTP/1.1 (loopback,
no TLS, so no HTTP/2 multiplexing), and Chrome allows only ~6 connections per
origin shared across ALL tabs. Each open mar tab permanently consumes one slot
with its SSE stream; at ~6 tabs the pool is exhausted and every new request
(navigation, CSS, favicon) stalls until a tab closes.

Fix options (best-first for a single-user local tool):
- Replace the held-open SSE with short polling of a version endpoint:
  GET /events/version returns the SQLite data_version integer; client polls
  every ~2s with a normal request that completes in ms and never holds a slot;
  reload when the number changes. Simplest, fully removes held connections, no
  new deps, keeps the single-binary/no-build property. (recommended)
- Or: one shared EventSource across tabs via SharedWorker/BroadcastChannel
  (N tabs -> 1 connection). Keeps push, more JS.
- Or: serve over HTTP/2 via a localhost TLS cert so SSE multiplexes (browser
  cert warnings, heavier).

Also (minor, found in the same HAR): GET /favicon.ico is served the index HTML
by the catch-all `GET /` route (Content-Type text/html). Add a favicon route
(204 or a real icon) so it stops matching `/` and competing for a slot.
